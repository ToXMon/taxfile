TOPIC A: New Jersey State E-File Requirements

Bottom Line for TaxFile: Because TaxFile generates only fillable PDFs and does not electronically transmit returns to NJ, it is NOT REQUIRED to register as an NJ e-file software provider, submit a Letter of Intent, undergo ATS testing, or obtain any NJ-specific e-file credentials. The requirements below apply only if/when TaxFile adds direct e-filing capability.

A.1 NJ Division of Taxation E-File Requirements for Software Providers

What Does NJ Require of Tax Software Providers?

Software vendors that electronically file returns with New Jersey must meet the following requirements [REQUIRED if e-filing]:

  • Complete a Letter of Intent (LOI) for each unique product offered
  • Undergo Assurance Testing System (ATS) testing and certification
  • Implement system security measures protecting taxpayer information online, offline, at rest, and in transit
  • Promptly report any data breaches or security incidents to NJDORES and the NJ Attorney General's Office
  • Include mandatory consent language for electronic filing (varies by software type: DIY, Professional, or Business)
  • Include specific URLs for refund processing and taxes-due information
  • Include driver's license or ID card with e-file returns (though NJ will not reject if missing)
  • Require desktop users to download updates if filing 10+ days after a production release

Is There a Separate NJ EFIN or Equivalent?

No. New Jersey does not issue a separate state EFIN. Tax preparers who are approved IRS Electronic Return Originators (EROs) are automatically recognized by NJ. Per the NJ E-File Mandate FAQ:

"Tax preparers, whether located in New Jersey or outside of the State, are not required to register for the New Jersey E-File Program provided that they are approved Electronic Return Originators (ERO) with IRS. The Division of Revenue and Enterprise Services receives their registration information from the IRS."

However, software vendors (as distinct from tax preparers) must still complete the LOI process and obtain IRS ATS certification before NJ will accept their test submissions.

NJ Electronic Filing Specifications and Schemas

  • NJ participates in the Modernized e-File (MeF) program for Individual, Partnership, Corporate, and Fiduciary returns
  • Access to NJ MeF schemas is granted via a secure state web portal after LOI acceptance
  • For bulk filing (WR-30, NJ-927, W-2, 1099), files must follow NJ-specific file layout formats
  • Form 1099-NEC has a unique NJ format: State Income Tax Withheld must appear in positions 723-734, right-justified, zero-filled

Does NJ Participate in the Fed/State Program?

Yes. New Jersey participates in the federal-state Modernized e-File (MeF) program. Covered return types:

Can You Transmit Both Federal and State Returns Together?

Yes. Under the MeF program, the federal return and NJ state return can be transmitted together in a single submission. The state return is included as part of the federal MeF submission package.

Technical Integration Requirements

  • Software must support IRS MeF transmission protocols
  • NJ-specific schema must be incorporated (accessed after LOI acceptance)
  • No separate NJ transmission infrastructure needed when using MeF — state return rides with federal
  • Separate transmission only required for bulk filings (WR-30, NJ-927, informational returns) which use a different channel (Axway)

TOPIC B: Data Breach Notification Requirements

Critical Finding: TaxFile's current architecture — plaintext JSON storage with no encryption at rest, deployed on Akash Network — has zero protection under NJ law's encryption safe harbor. Any unauthorized access to stored data constitutes a reportable breach if the data qualifies as "personal information."

B.1 NJ Data Breach Notification Act (N.J.S.A. 56:8-161 et seq.)

Statutory Framework

StatutePurpose
N.J.S.A. 56:8-161Definitions (personal information, breach of security, encryption safe harbor)
N.J.S.A. 56:8-162Methods of destruction of personal information
N.J.S.A. 56:8-163Disclosure of breach of security to customers (notification requirements)
N.J.S.A. 56:8-164Additional requirements (expanded by P.L.2019, c.95)
N.J.S.A. 56:8-165Penalties — violations classified as Consumer Fraud Act violations
N.J.S.A. 56:8-166Severability
N.J.A.C. 13:45F-5.2Administrative code implementing regulations

Current 2024-2025 Requirements

The law applies to: "Any business that conducts business in New Jersey, or any public entity that compiles or maintains computerized records that include personal information." (N.J.S.A. 56:8-163(a))

What Triggers Notification Obligation?

Notification is triggered when both conditions are met:

  1. A breach of security occurs (unauthorized access to electronic files containing personal information that compromises security, confidentiality, or integrity) (N.J.S.A. 56:8-161)
  2. The personal information was, or is reasonably believed to have been, accessed by an unauthorized person (N.J.S.A. 56:8-163(a))

Exception: No disclosure required if the business establishes that misuse of the information is not reasonably possible, provided such determination is documented in writing and retained for five years (N.J.S.A. 56:8-163(a))

Timeline for Notification

RecipientTimelineStatute
NJ Division of State PoliceBEFORE notifying affected individualsN.J.S.A. 56:8-163(a)
Affected NJ residents"Most expedient time possible and without unreasonable delay", consistent with law enforcement needs and reasonable time to determine scope and restore integrityN.J.S.A. 56:8-163(a)
Consumer Reporting Agencies (if 1,000+ persons)Without unreasonable delay — timing, distribution, and content of noticesN.J.S.A. 56:8-163(a)
Law Enforcement DelayNotification delayed if law enforcement determines it will impede investigation and requests delay; resumes when law enforcement clearsN.J.S.A. 56:8-163(c)
Important: There is no fixed number-of-days deadline in NJ law (unlike some states with 30/60/72-hour requirements). "Most expedient time possible without unreasonable delay" is a facts-and-circumstances standard. However, reporting to the Division of State Police must occur before individual notifications go out.

Recent Amendments (P.L.2019, c.95 / S.B. 424)

Signed May 10, 2019; effective September 1, 2019. Key changes:

  • Expanded "personal information" definition to include online account information: "user name, email address, or any other account holder identifying information, in combination with any password or security question and answer that would permit access to an online account"
  • Electronic notification permitted for breaches involving only username/password combinations (no other personal information)
  • Username/password-only breach notices must direct customers to: (1) promptly change password/security Q&A, and (2) protect other online accounts using the same username

No comprehensive federal breach notification law exists. There is no federal equivalent to state breach notification statutes that broadly requires notification to affected individuals. However, several federal frameworks may apply depending on the business's classification:

B.3 What Constitutes a 'Breach' Under NJ Law for Tax Data

Definition of 'Personal Information'

Under N.J.S.A. 56:8-161, personal information means an individual's first name or first initial and last name linked with one or more of the following:

  1. Social Security number
  2. Driver's license number or State identification card number
  3. Account number or credit/debit card number, in combination with any required security code, access code, or password permitting access to a financial account
  4. Username, email address, or other account holder identifying information, in combination with any password or security question/answer permitting access to an online account (added by P.L.2019, c.95)

Also includes dissociated data that would constitute personal information if linked, if the means to link the dissociated data were accessed in connection with the access to the data itself.

Exclusion: Publicly available information from government records or widely distributed media is excluded.

Does SSN + Name Constitute Covered Data?

YES — unconditionally. Name + SSN is the quintessential example of "personal information" under NJ law. This is the first and most clearly covered data element combination in the statute.
TaxFile-Specific Critical Analysis: According to the TaxFile Security Audit (2026-03-26), TaxFile currently stores only masked SSNs (format: XXX-XX-1234), not full SSNs. A masked SSN (showing only last 4 digits) likely does NOT constitute a Social Security number under the statute, because it cannot be used for identity theft or unauthorized purposes in the same way as a full SSN. However, this analysis is not certain — no NJ case law directly addresses whether a masked SSN qualifies. If TaxFile adds full SSN storage (which is required for actual tax filing), name + full SSN = unambiguously covered personal information.

Does Plaintext JSON Storage Affect Breach Determination?

YES — critically. The encryption safe harbor in N.J.S.A. 56:8-161 provides:

"Breach of security" means unauthorized access to electronic files, media, or data containing personal information that compromises the security, confidentiality, or integrity of personal information when access to the personal information has not been secured by encryption or by any other method or technology that renders the personal information unreadable or unusable.

This means:

  • If data is encrypted (rendered unreadable/unusable): Unauthorized access does NOT constitute a "breach of security" — no notification required
  • If data is NOT encrypted (TaxFile's current plaintext JSON): Any unauthorized access to files containing personal information automatically constitutes a "breach of security" — notification IS required
TaxFile Risk: Plaintext JSON storage on Akash Network means TaxFile has no encryption safe harbor protection whatsoever. Even if an attacker gains access to a server but does not exfiltrate data, the mere unauthorized access to unencrypted personal information triggers the notification obligation.

Does Unauthorized Access to a Server Constitute a Breach Even Without Exfiltration?

YES. The NJ statute defines breach of security as "unauthorized access to electronic files, media, or data containing personal information that compromises the security, confidentiality, or integrity of personal information." The trigger is access, not exfiltration. If an unauthorized person can read the data (which is trivially true for plaintext JSON), the security and confidentiality of that data have been compromised, regardless of whether the data was copied or downloaded.

B.4 Penalties for Failing to Notify

Classification as Consumer Fraud Act Violation

N.J.S.A. 56:8-165 provides:

"It shall be an unlawful practice and a violation of [the Consumer Fraud Act, N.J.S.A. 56:8-1 et seq.] to willfully, knowingly or recklessly violate N.J.S.A. 56:8-161 through 164."

This means breach notification violations are prosecuted under the New Jersey Consumer Fraud Act (CFA), which provides:

Penalty TypeAmountSource
Civil penalty — first offenseUp to $10,000N.J.S.A. 56:8-11
Civil penalty — subsequent offenseUp to $20,000N.J.S.A. 56:8-11
Treble damages (private action)3x actual damagesN.J.S.A. 56:8-19
Attorney's fees and costs (private action)Reasonable attorney's fees + costsN.J.S.A. 56:8-19

Attorney General Enforcement Powers

  • The NJ Attorney General can bring civil enforcement actions under N.J.S.A. 56:8-11
  • Can seek injunctive relief to stop ongoing violations
  • Can assess civil penalties ($10,000/$20,000 per violation)
  • The NJ Division of State Police receives breach reports for investigation and potential referral to law enforcement (N.J.S.A. 56:8-163(a))

Criminal Penalties

No specific criminal penalties are enumerated in the NJ breach notification statute itself (N.J.S.A. 56:8-161 et seq.). However, if the breach involves criminal conduct (e.g., unauthorized access to computer systems under NJ computer crimes statute N.J.S.A. 2C:20-25 et seq.), separate criminal charges could apply to the perpetrator — not to the business for failing to notify.

Private Right of Action

YES. The NJ general breach notification statute allows for a private right of action. Affected individuals can sue for:

  • Treble (3x) actual damages
  • Reasonable attorney's fees and costs

This makes NJ one of the more plaintiff-friendly breach notification regimes in the country.

B.5 IRS Requirements for Reporting Breaches Involving Taxpayer Data

Does the IRS Require Notification of Breaches Involving Taxpayer Data?

TaxFile-Specific Answer: NO — not currently. IRS breach reporting requirements under Publication 1345 apply only to Authorized IRS e-file Providers. TaxFile, which generates only PDFs and does not transmit returns electronically, is explicitly exempt from Pub 1345 requirements as a "Software Developer that does not engage in any IRS e-file activity other than software development." (Pub. 1345, Rev. 12-2025)

For context, here is what would apply if TaxFile were an Authorized IRS e-file Provider:

IRC Section 6103 — Applicability Analysis

IRC Section 6103 governs confidentiality and disclosure of "returns" and "return information." Key analysis for TaxFile:

  • IRC 6103 primarily restricts the IRS and its agents from disclosing taxpayer information. The statute creates obligations on the government and authorized recipients, not on unrelated third parties.
  • "Returns" and "return information" under 6103 specifically refer to information filed with or obtained from the IRS. Tax data that a user enters into a private software application (without ever being transmitted to IRS) may not qualify as "return information" under 6103.
  • However, if TaxFile were an Authorized IRS e-file Provider, it would receive taxpayer data in connection with IRS e-file and 6103's restrictions on disclosure would apply. Since TaxFile is not an e-file provider, this chain does not exist.
  • IRC 6103(n) provides that any person who is convicted of unauthorized inspection of returns or return information shall be fined or imprisoned, but this applies to persons who obtained the information from the IRS or its agents.
Caveat: This analysis is not legal advice. Whether IRC 6103 applies to a third-party software company that stores taxpayer data (without IRS authorization) is an untested legal question. No published case law directly addresses this scenario. If TaxFile begins e-filing, 6103 would unambiguously apply.

Highest-Priority Risks for TaxFile

  1. Plaintext JSON storage = zero safe harbor. If TaxFile ever stores full SSNs (required for actual tax filing), any unauthorized server access triggers NJ breach notification obligations — even without exfiltration. Akash's decentralized architecture increases the attack surface for unauthorized access.
  2. Zero authentication on all API endpoints. The security audit confirmed no authentication on any endpoint, making unauthorized access trivially easy and virtually guaranteed to occur if exposed to the internet.
  3. Private right of action + treble damages. NJ is one of the most plaintiff-friendly breach notification states. A single breach affecting even a small number of NJ residents could result in significant litigation exposure.
  4. Consumer Fraud Act penalties. Willful, knowing, or reckless failure to notify exposes TaxFile to $10,000-$20,000 per violation civil penalties from the NJ AG, on top of private litigation.
